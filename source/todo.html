Outstanding Issues

This pages summarizes the known outstanding issues and todo notes currently found in the FHIR specification. Most of these issues (thought not all) are noted through out the specification in the appropriate location. Many of these questions are under investigation during the current connectathon cycle parallel to the ballot.

Note that adding additional resources is an ongoing process - see notes here (bottom of the page).

Section Issues
Publication Format
  • fix single page form
General
  • Develop Relax NG, RDF, and eCore definitions of the resources
  • Should a resource identify which profile(s) it conforms to? (why?) (related discussion: rules of handling and version dependency)
  • What consistent life cycle patterns do we need for resources?
  • There is still a lack of clarity around how patient linking/unlinking/merging/unmerging should be handled
Reference Implementations
  • Delphi reference implementation has dependencies on unpublished code
  • Consider other publication options for C# and Java (e.g. maven?)
Examples
  • Examples continue to develop and be fleshed out. Several are marked with links that need fixing - these correspond to further additional planned examples
Data Types
  • The way Attachment.uri works with relative references needs to be properly described
  • Does language ever matter on Coding.display?
Extensibility
  • point to actual registry once it's up and running
  • How to manage published extension life cycle (i.e. deprcation)
Narrative
  • Formally encode cosntraints in schematron
Security & Security Event
  • This is under active investigation through connectathons running parallel to the ballot process, and further work is expected on this section. Interested parties should track the joint security / FHIR work in this area
  • Profile for OAuth based login
  • Do we need an element for Media Id (as opposed to type)?
Codes & Terminologies
  • The named system list needs expansion
Conformance / Profile
  • Add flag for accept unknown content
  • Need to add back to the conformance resource about how creates and ids work for a system
  • Need to avoid imposing requirement to keep version history while still supporting versioning as much as possible
  • Do we need something more formal for security capabilities?
  • Does a system need to be able to enumerate profiles that it does support?
  • Need to do more to flesh out messaging dynamic model?
  • May need profile id as well if profiles can define message events?
  • Where do search parameters belong - profile or conformance, or both?
  • How does the profile slicing discriminator work with resource references
  • How are bundles managed in profiles?
Care Plan
  • Isn't <performer> redundant when considering <participant> that was defined before
Condition
  • The physician states (in the EHR) that fever could be caused by another (bacterial) infection.How do you express such an (important!) fact with FHIR
  • The difference between "due-to" and "follows" is very unclear in the current documentation
  • Discuss the applicability of assessing stages
  • due-to: How to differentiate from follows?
  • Condition.age = is a range needed?
  • Condition.stage - narrow
Coverage
  • Need a definition for Coverage.group
Device Capabilities
  • What communications protocol is used with device capabilities / device log resources?
Diagnostic Report
  • Should performer be allowed to be an organization?
  • DiagnosticReport.requestDetail.encounter / DiagnosticReport.requestDetail.requestOrderId / DiagnosticReport.requestDetail.receiverOrderId - Question whether these are 80%
  • Reference Range Meaning - where can appropriate codes come from?
Document
  • Confidentiality? Language? Consent? Signatures
  • Need to extend this to support documents with other types of subjects. Also rationalize with encounter & context elements
  • Document.section.content: This may need to be 0..*. May need to consider adding text here to account for missing resources
Encounter
  • Encounter.type: Why is this merely a CodeableConcept and not Resource (any)?
Group
  • We need determiner codes that indicate we're talking about a group even when we haven't specified a specific quantity
List
  • does Loinc define useful codes for list.type?
Location
  • Shouldn't that be "owns or manages"? Why introduce the fact that an organization provides services here? If this is a home location, you wouldn't mention the organization(s) providing services there?
Medication
  • Is it necessary to assign an identifying code to a medication record so that a system's list of medication resources can also be used as a coding system?
Medication Administration
  • Expand notes on relationship between quantity and rate
  • Explain what happens if rate is not continuous
Medication Prescription
  • reasonForPrescribing: Why can't this be Resource (any)
  • doseQuantity: Cardinality is not sufficient: 25ug salmeterol + 250ug fluticasone can only be coded with 275ug...
  • <rate> and <maxDosePerPeriod> are not sufficient for this specific patient: a max dose is higher than a daily amount and rate is intended for (e.g. infuus-)rates, whilst prescribed amount needs to be expressed...
Messaging
  • Describe some use cases
  • Consider changing the code to Coding to support profile-defined events
  • Message.effectiveTIme: this is not 80/20? Also, it should really be in the domain resources, not tucked away here
Procedure
  • Procedure.indication: Why can't this be Resource (any) AND String?
  • Should there be a separate entry for duration? Currently this is inferred from date if represented as a Period
  • Indication is currently a string. What to do if you want to indicate that the procedure is because of a Condition? You could use relatedItem, but is it confusing to have indication in multiple places.
  • Should we support a careplan as followup?
Query
  • How are errors processed?
